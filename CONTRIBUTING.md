# Contributing to GrillKit

Thank you for your interest in contributing. Read [ARCHITECTURE.md](ARCHITECTURE.md) before changing application code.

## Getting Started

1. Fork the repository
2. Clone your fork: `git clone https://github.com/yourusername/grillkit.git`
3. Create a branch: `git checkout -b feature/your-feature-name`

## Development Setup

```bash
uv sync --extra dev
uv run pytest
uv run uvicorn app.main:app --reload
```

Docker (recommended for a production-like run):

```bash
docker compose up --build
```

## How to Contribute

### Reporting Bugs

- Check for an existing issue first
- Steps to reproduce, expected vs actual behavior
- OS, Python version, GrillKit version (`YYYY.M.d` from [CHANGELOG.md](CHANGELOG.md))
- Logs or screenshots when relevant

### Suggesting Features

- Open an issue using the feature request template
- Describe the use case and proposed behavior

### Adding Questions

Questions live in YAML under `data/questions/{language}/{level}/{category}.yaml`.

Example (`data/questions/python/junior/data-structures.yaml`):

```yaml
category: "Data Structures"
language: "python"
level: "junior"

description: "Lists, tuples, dicts, and sets"

questions:
  - id: "ds-001"
    type: "knowledge"
    difficulty: 1
    tags: ["list", "tuple"]
    question:
      text: "What is the difference between a list and a tuple?"
      code: null
    follow_ups:
      - "When would you choose a tuple over a list?"
    expected_points:
      - "Lists are mutable; tuples are immutable"
```

Guidelines:

- Match the schema used in existing banks (`category`, `language`, `level`, `questions`)
- Use clear wording; set `difficulty` 1–5 within the level
- File name = category slug (e.g. `data-structures.yaml`)
- Validate YAML syntax before opening a PR

Note: `follow_ups` and `expected_points` are stored for future use; follow-ups in interviews are generated by the AI today.

### Code Contributions

1. Add tests for new behavior
2. Run quality checks (see below)
3. Update [README.md](README.md), [ARCHITECTURE.md](ARCHITECTURE.md), or [CHANGELOG.md](CHANGELOG.md) when behavior or layout changes
4. New Python files: Apache-2.0 copyright header (see existing modules in `app/`)
5. Keep commits focused; use [conventional commits](https://www.conventionalcommits.org/)

Example commit messages:

```
feat(api): add interview timer endpoint
fix(services): persist score after WebSocket completion
docs: update README roadmap
test(repositories): cover mark_completed
```

## Code Standards

```bash
uv run ruff check --fix .
uv run ruff format .
uv run mypy .
uv run pytest
```

- Type hints on public APIs; `mypy --strict` must pass
- Follow layer boundaries in [ARCHITECTURE.md](ARCHITECTURE.md): `api/` → `services/` → `domain/` / `repositories/`

## Pull Request Process

1. Rebase or merge latest `main`
2. All checks above pass
3. Fill out the [pull request template](.github/pull_request_template.md)
4. Request review
